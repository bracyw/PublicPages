Confluence Consolidation & Change Recommendations

Software docs live in two homes on our Confluence: the shared car trees (NER25, NER27 …), where every team documents each build cycle and software’s current work is genuinely alive — and a top-level “Software” section, the team’s old front door, which froze in 2023–24 and routes to none of it. We plan annually, around the car — this proposal makes the space say so: one home under the year’s tree, the frozen section archived entirely, a small Shared Resources folder that walks forward each year, and one convention so it stays clean. Nothing has been changed yet.

01

The problem, briefly

  • The car trees work — keep them. The space gives each car generation a shared tree (NER25, NER27) where every team files its cycle work. In NER27, Mechanical and Electrical each gathered their whole team into one folder; beside those sit “Embedded Software” — the firmware branch, where the 27A cycle is being built right now — and FinishLine’s week-old “Web Development” branch. Software has filed its cycle docs this way since NER24. And NERxx is really the season’s tree, not just the chassis — Admin, Club Initiatives, and the year’s meeting notes live there too — so it fits every subteam, including the ones that don’t ship on the car.
  • The team’s front door froze. The top-level “Software” section — the evergreen home for onboarding, FAQ, learning resources, FinishLine, and Product — has 33 of its 39 pages untouched since 2024. The onboarding root names heads from two cycles ago, routes to an archived Slack channel, and offers paths for only two of the four subteams; the org page describes a team merge that was later abandoned; and nothing links the car trees. Browsing from “Software” reaches none of the live work.
  • The car branches still speak pre-Charter. “Embedded Software” is the old umbrella — its own 25 page defines it as “Firmware – Electrical Team + Applications Software – Software Team,” an org the Charter replaced. Application Software’s docs sit buried inside it, and FinishLine’s branch is titled “Web Development.” A newcomer can’t map the Charter’s four subteams onto the space.
  • Cycles supersede but never archive, and titles mislead. Embedded Software 24 sits unarchived beside 25, an empty Odyssey 24A beside a real 25A, and FinishLine’s old branch holds six roadmap cycles side by side. “Web Dev Onboarding” is FinishLine-only; two stale 2024 Launchpads sit in the Software section while the live one (Firmware Launchpad 26-27) lives under NER27; one page has sat in draft since Sep 2024.
The goal is a space shaped like the way we actually work: annual planning, around the car. Years in titles are fine — the problem is the mix, and a front door that goes nowhere. Put everything in the year’s tree, name the branches after the org that exists, keep one small Shared Resources folder that moves forward at handoff, and archive superseded pages instead of leaving them as siblings.

Recommendation

Three options that stack — each includes everything to its left. B is the ask.

A

Housekeeping

Archive the frozen layer (list in §03). The two README-linked entry points — the org page and the onboarding root — get a one-line tombstone (“archived — current work lives under NER27”) before archiving, so inbound links degrade gracefully. Retitle the mislabeled pages. No new structure.
Effort ~1 sitting per area · When this week
B

Go fully annual — one home

The ask
A, plus: everything lives under NER27 → Software — the one-folder-per-team shape Mechanical and Electrical already use — with a branch per subteam (Firmware · App Software · FinishLine · Product), onboarding per subteam inside each, and a landing page carrying the year’s head, leads, and pick-a-subteam router (a fresh snapshot each cycle, like today’s Embedded Software page). Members-only like the rest of NER27 — no parallel public tree to maintain. At each handoff the small Shared Resources folder (env setup, system docs) moves — not copies — into the new year’s tree, so page IDs and inbound links survive the walk. NERxx is the club’s clock — the season, not just the chassis — so FinishLine’s and Product’s pages belong there as much as firmware’s.
Effort a few hours per subteam · When before fall onboarding
C

Align with the 26-27 Vision

B becomes the skeleton for the Vision’s restructured onboarding (software-wide intro + technical “droplets,” authored Confluence-first). Mechanical and electrical are already rebuilding their onboarding for 26-27 — this matches that pace.
Effort the fall onboarding project · When Sep 2026

The convention

Two kinds of page Cycle pages (onboarding flows, rosters, roadmaps, project docs) carry the year — <Subteam> <Topic> 26-27 — and archive with their tree. Evergreen pages (setup guides, system docs) never carry a year; they live in Shared Resources and move forward into the new year’s tree at handoff. The year-stamp is the archive bit — no judgment calls. The cost of mixing the two is already visible: Embedded Software 25 opens by pointing readers back at the 24A, 22A, and 17D docs for design decisions that still apply.
Titles say subteam + topic A title must be correct out of context, because search shows it out of context. No more “Web Dev Onboarding.”
Two archive triggers Supersede: publishing a successor means archiving the predecessor in the same sitting — the missing habit that produced six side-by-side FinishLine roadmaps. Retire: when the car retires, its whole Software branch archives as a unit (Shared Resources has already walked out). Two live car trees is normal — 25A competed while 27 was designed; three is rot. Never delete: archived pages leave the tree and default search, but direct links still resolve.
Names are fine in cycle pages A year-stamped page is a snapshot — name the year’s head and leads freely. Only Shared Resources pages say roles (“the App Software head”), because they outlive rosters.
Durable links point at Shared Resources Repo READMEs and pinned messages never deep-link cycle pages (new year, new ID). Link Shared Resources pages — their IDs survive the annual walk — or write the path as text: NER2X → Software → App Software → Onboarding.
Per-cycle ritual At handoff: create the new year’s Software branch, move Shared Resources in, write the year’s cycle pages, archive on the two triggers. ~1 hour per subteam, once a year — this proposal never has to be written again.
